# 前端/后端同步到服务器 — 最终确认 SOP

> **核心原则：** 必须先在本机完成 push，再在服务器上执行任何操作。服务器上的 `git fetch` 必须在 push 之后执行，否则会拉到旧代码。

---

## 一、前置条件（首次或 Nginx 变更时检查一次）

在服务器上执行，确认 Nginx 指向正确目录：

```bash
grep -r "root.*plc-sim" /etc/nginx/
```

**要求：** www.plc-sim.com 对应的 `root` 必须是 `/var/www/plc-sim/polo/frontend`。  
若是 `/var/www/plc-sim/frontend/dist` 或其它路径，需修改 Nginx 配置并 `sudo nginx -t && sudo nginx -s reload`。

---

## 二、标准操作流程（每次改代码后）

### 阶段 A：本机（必须全部完成后再去服务器）

| 步骤 | 命令 | 确认 |
|-----|------|------|
| A1 | Windows:`cd E:\Project\V2.13` Mac: cd "/Users/huanglijun/Desktop/Polo/Cursor   project/V2.13"| 进入项目根目录 |
| A2 | `git add -A` | 所有改动已加入暂存区 |
| A3 | `git status` | 确认列表中有你改的文件（如 `src/App.tsx`） |
| A4 | `git commit -m "你的修改说明"` | 看到 `1 file changed` 或类似输出 |
| A5 | `git push origin main` | 看到 `main -> main` 或 `Everything up-to-date`（无报错） |
| A6 | `git log -1 --oneline` | 记下最新提交的 hash 和说明（如 `968a22a 标头图标改为齿轮`） |

**⚠️ 必须确认 A5 的 push 已成功，再执行阶段 B。若 push 失败，先解决网络或权限问题。**

---

### 阶段 B：服务器（必须在阶段 A 全部完成之后执行）

SSH 登录服务器后，**严格按顺序**执行：

| 步骤 | 命令 | 确认 |
|-----|------|------|
| B1 | `cd /var/www/plc-sim/polo` | 进入仓库根目录 |
| B2 | `git fetch origin` | 无报错 |
| B3 | `git log -1 origin/main --oneline` | **必须**与阶段 A6 的提交一致（hash 和说明相同）。若不一致，说明 push 未生效或 fetch 到的是旧数据，需回到本机重新 push 或检查 remote |
| B4 | `git reset --hard origin/main` | 输出 `HEAD is now at xxxxx 你的提交说明`，且与 A6 一致 |
| B5 | `npm ci` | 无报错，看到 `added xxx packages` |
| B6 | `npm run build` | 无报错，看到 `vite build` 和 `dist/` 生成 |
| B7 | `sudo rsync -a --delete dist/ /var/www/plc-sim/polo/frontend/` | 无报错 |
| B8 | `sudo chown -R www-data:www-data /var/www/plc-sim/polo/frontend` | 无报错 |
| B9 | `pm2 restart plc-sim-api` | 若有后端改动则执行；仅前端改动可省略 |

---

### 阶段 C：验证

| 步骤 | 操作 | 确认 |
|-----|------|------|
| C1 | 打开 https://www.plc-sim.com | 页面能正常打开 |
| C2 | 按 **Ctrl+Shift+R**（或 F12 → 右键刷新 →「清空缓存并硬性重新加载」） | 强制刷新，避免读缓存 |
| C3 | 检查页面内容 | 图标、样式、功能是否与本地修改一致 |

## 三、完整命令清单（可直接复制）

### 本机（PowerShell）

```powershell
cd E:\Project\V2.13
git add -A
git status
git commit -m "你的修改说明"
git push origin main
git log -1 --oneline
```

### 服务器（Bash，在 push 成功之后执行）

```bash
cd /var/www/plc-sim/polo
git fetch origin
git log -1 origin/main --oneline
git reset --hard origin/main
npm ci
npm run build
sudo rsync -a --delete dist/ /var/www/plc-sim/polo/frontend/
sudo chown -R www-data:www-data /var/www/plc-sim/polo/frontend
pm2 restart plc-sim-api
```

### 验证

浏览器打开 https://www.plc-sim.com，按 **Ctrl+Shift+R** 强刷。

## 四、一键部署（与手动步骤等价）

本机 push 完成后，在服务器上执行：

```bash
cd /var/www/plc-sim/polo
./deploy/scripts/deploy-polo.sh
```

脚本会执行：`git fetch` → `git reset --hard origin/main` → `npm ci` → `npm run build` → rsync → chown → pm2 restart。  
**同样要求：本机必须先 push，再运行此脚本。**

---

## 五、常见错误与对应处理

| 错误 | 表现 | 处理 |
|------|------|------|
| **服务器先 fetch，本机后 push** | 服务器 `git log -1 origin/main` 是旧提交，构建出旧页面 | 本机先 `git push origin main`，确认成功后再在服务器执行 `git fetch` 和后续步骤 |
| **Nginx root 指错** | 部署了但网页不变 | 检查 `grep -r "root.*plc-sim" /etc/nginx/`，www 的 root 必须是 `/var/www/plc-sim/polo/frontend` |
| **先 build 再 fetch/reset** | 构建用的是旧代码 | 顺序必须是：fetch → reset → npm ci → build → rsync |
| **未强刷浏览器** | 服务器已更新，本地仍看到旧版 | 使用 **Ctrl+Shift+R** 或无痕模式打开 |
| **git add 漏了文件** | commit 里没有改动的文件 | 用 `git add -A` 或 `git add src/App.tsx` 等明确指定 |

## 六、检查清单（网页未更新时逐项排查）

1. [ ] 本机 `git push origin main` 已成功执行？
2. [ ] 服务器 `git log -1 origin/main --oneline` 与本机最新提交一致？
3. [ ] 服务器 `git reset --hard origin/main` 的输出 hash 与上面一致？
4. [ ] Nginx 的 root 是 `/var/www/plc-sim/polo/frontend`？
5. [ ] 浏览器已按 **Ctrl+Shift+R** 强刷？
6. [ ] 可尝试无痕模式打开 https://www.plc-sim.com 验证

---

按此 SOP 操作，前端/后端改动即可稳定同步到线上。**关键：先 push，再在服务器上 fetch 和部署。**
