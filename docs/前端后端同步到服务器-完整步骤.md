# 前端/后端同步到服务器 — 正确 SOP

## 一、问题出在哪里（本次同步的根因）

| 问题 | 表现 | 根因 | 解决 |
|------|------|------|------|
| **1. Nginx 指错目录** | 部署了但网页样式/图标不变 | Nginx 的 `root` 指向了 `/var/www/plc-sim/frontend/dist` 等旧路径，而实际发布目录是 `/var/www/plc-sim/polo/frontend` | 把 www 站点的 `root` 改为 `/var/www/plc-sim/polo/frontend`，并 `nginx -s reload` |
| **2. 服务器代码没更新** | 本机已 push，服务器 `git pull` 仍显示 “Already up to date” | 服务器本地分支未与远程同步（或 remote 地址不一致），pull 没有真正拿到最新提交 | 用 `git fetch origin` + `git reset --hard origin/main` 强制与远程一致，再构建 |
| **3. 构建用的是旧代码** | 即使 pull 了，网页仍旧 | 先执行了 `npm run build` 再 pull，或 pull 未生效就 build，导致 dist 是旧代码 | 严格顺序：**先更新代码（fetch+reset 或 pull）→ 再 build → 再 rsync** |
| **4. 浏览器缓存** | 服务器已更新，自己电脑上看还是旧版 | JS/CSS 被强缓存 | 使用 **Ctrl+Shift+R** 或「清空缓存并硬性重新加载」 |

---

## 二、正确 SOP（标准操作流程）

### 本机（每次改完前端或后端后）

```powershell
cd E:\Project\V2.13
git add -A
git commit -m "修改说明"
git push origin main
```

**要求：** 看到 push 成功（无报错）；若显示 “Everything up-to-date”，说明当前分支已和远程一致，可继续。

---

### 服务器（SSH 登录后，严格按顺序执行）

```bash
cd /var/www/plc-sim/polo

# 1. 先让代码与远程一致（避免 pull “Already up to date” 但实际未更新）
git fetch origin
git reset --hard origin/main

# 2. 安装依赖并构建
npm ci
npm run build

# 3. 把构建产物发布到 Nginx 使用的目录
sudo rsync -a --delete dist/ /var/www/plc-sim/polo/frontend/
sudo chown -R www-data:www-data /var/www/plc-sim/polo/frontend

# 4. 若有后端改动，重启 API
pm2 restart plc-sim-api
```

---

### 验证

- 打开 https://www.plc-sim.com
- 按 **Ctrl+Shift+R** 强制刷新，确认页面/图标已更新。

---

## 三、一键部署（与上述 SOP 等价）

服务器上若已配置好部署脚本，可直接：

```bash
cd /var/www/plc-sim/polo
./deploy/scripts/deploy-polo.sh
```

脚本内部会：`git fetch` + `git reset --hard origin/main`、`npm ci`、`npm run build`、rsync 到 `polo/frontend`、chown、以及 pm2 重启。与第二节手动步骤一致。

---

## 四、网页仍未更新时的排查清单

按顺序检查，处理第一次碰到的问题即可：

1. **Nginx root 是否正确**  
   `grep -r "root.*plc-sim" /etc/nginx/`  
   www 站点必须是 `root /var/www/plc-sim/polo/frontend;`。若是 `frontend/dist` 或别的路径，改为此路径后 `sudo nginx -t && sudo nginx -s reload`。

2. **服务器代码是否最新**  
   `git log -1 origin/main --oneline` 是否为本机刚 push 的那条提交。若不是，执行 `git fetch origin` 和 `git reset --hard origin/main` 后重新 build 和 rsync。

3. **是否先更新代码再构建**  
   顺序必须是：更新代码 → `npm run build` → rsync → chown。不能先 build 再 pull。

4. **浏览器是否强刷**  
   使用 **Ctrl+Shift+R** 或 F12 → 右键刷新 →「清空缓存并硬性重新加载」。

---

## 五、约定（避免再次出错）

- **唯一前端发布目录：** `/var/www/plc-sim/polo/frontend`（Nginx 的 root 必须指向此处）。
- **代码更新方式：** 服务器上用 `git fetch origin` + `git reset --hard origin/main`，保证与远程 main 一致后再构建。
- **发布顺序：** 更新代码 → `npm ci` → `npm run build` → rsync → chown →（有后端改动时）pm2 restart。

按此 SOP 操作，前端/后端改动即可稳定同步到线上。
