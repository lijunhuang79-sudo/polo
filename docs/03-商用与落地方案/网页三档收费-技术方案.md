# 网页三档收费 — 技术方案

> **业务方案**：免费档（仅开放 2 个典型场景示例）+ 基础版 9.9 元（永久本地生成）+ AI 周卡 19.9 元（7 天有效，且须在基础版已开通后方可激活）。本文档描述技术实现方案，不涉及具体代码实现细节。

---

## 一、方案概述

### 1.1 三档权益

| 档位 | 价格 | 权益 | 交付方式 |
|------|------|------|----------|
| **免费档** | 0 元 | 仅可打开并使用「典型场景示例」中的前两项：**启保停控制**、**延时启动**；其余场景示例及「本地生成」「AI 智能生成」均不可用 | 无需支付，默认即为免费档 |
| **基础版** | 9.9 元 | 永久使用「本地生成」模式；典型场景示例全部开放 | 支付成功后下发永久授权码 |
| **AI 周卡** | 19.9 元 | 7 天内使用「AI 智能生成」模式 | 支付成功后生成 7 天有效 token，以链接+二维码形式交付；**仅当基础版已开通时方可购买并激活** |

### 1.2 档位关系与约束

- **免费档**：未付费用户默认处于此档；只能使用 2 个固定场景示例（启保停控制、延时启动），不能自由输入描述、不能使用本地生成与 AI 生成。
- **基础版（9.9）**：购买并激活后，典型场景示例全部开放，且可使用「本地生成」；**不依赖**其他档位，可单独购买。
- **AI 周卡（19.9）**：**依赖基础版**。只有在「基础版已开通」的前提下，用户才能：
  - 发起 AI 周卡订单（购买 19.9）；
  - 激活/使用已购买的 AI 周卡 token。  
  若未开通基础版，则不允许购买 19.9，或购买后在使用时提示「请先购买并激活基础版 9.9 元」直至基础版激活后再开放 AI 能力。

### 1.3 使用关系小结

| 用户状态 | 典型场景示例 | 本地生成 | AI 智能生成 |
|----------|--------------|----------|-------------|
| 免费档 | 仅「启保停控制」「延时启动」2 项可点 | 不开放 | 不开放 |
| 已购基础版（9.9） | 全部开放 | 开放 | 不开放，入口提示「购买 AI 周卡」 |
| 基础版 + AI 周卡有效期内 | 全部开放 | 开放 | 开放 |

### 1.4 技术目标

- 网页端完成支付、领取权益、凭链接/二维码使用；免费档无需登录即可使用 2 个固定场景。
- 后端完成订单记录、授权与 token 生成与校验、**AI 周卡下单/激活时校验基础版是否已开通**。
- 前端根据「是否免费档」「基础版是否解锁」「AI token 是否有效」控制：场景示例展示数量、本地生成入口、AI 生成入口及 19.9 购买入口。

## 二、免费档与「典型场景示例」的约定

### 2.1 免费可用的两个场景

与现有 `constants.ts` 中 `SCENARIOS` 对应关系建议如下（按标题或固定索引均可，实现时二选一即可）：

| 场景标题 | 说明 | 免费档是否可用 |
|----------|------|----------------|
| 基础: 启保停控制 | 启动/停止按钮，自锁 | ✅ 可用 |
| 基础: 延时启动 | 延时 3 秒后电机启动，停止立即停 | ✅ 可用 |
| 其余所有场景 | 双次启动、星三角、正反转、红绿灯、车库门、混合罐、计数、电梯、PID 等 | ❌ 不可用，展示为「升级基础版解锁」或置灰 |

### 2.2 前端展示逻辑（建议）

- 免费档：在「典型场景示例」折叠区内，仅「启保停控制」「延时启动」两项可点击并填充到输入框；其余项置灰或隐藏，或展示锁图标 + 文案「升级基础版解锁全部场景」。
- 已激活基础版：所有场景示例均可点击；本地生成入口开放；AI 入口根据是否拥有有效 AI token 显示「购买 19.9 AI 周卡」或正常使用。
- 前端需持久化或每次校验「当前是否免费档 / 是否已激活基础版」，以便正确展示上述逻辑。

## 四、数据模型（建议）

### 4.1 订单表 `orders`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | 订单号（可兼作业务订单号） |
| product_type | enum | `basic` \| `ai_week` |
| amount | decimal | 金额（9.9 / 19.9） |
| pay_status | enum | pending \| paid \| refunded \| failed |
| pay_channel | string | 支付渠道订单号 |
| pay_time | datetime | 支付成功时间 |
| buyer_open_id / buyer_id | string | 支付渠道用户标识（若需） |
| created_at | datetime | 创建时间 |

### 4.2 基础版授权表 `basic_licenses`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | |
| order_id | FK | 关联订单 |
| license_key | string | 永久授权码（唯一，可展示给用户保存） |
| used_at | datetime | 首次使用时间（可选） |
| created_at | datetime | 发放时间 |

- 一个订单对应一条 `basic` 记录，一条记录对应一个永久 `license_key`。

### 4.3 AI 周卡 token 表 `ai_week_tokens`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | |
| order_id | FK | 关联订单 |
| token | string | 唯一 token（放在链接中，或生成二维码内容） |
| valid_from | datetime | 生效时间（通常=创建时间） |
| valid_until | datetime | 过期时间（创建时间 + 7 天） |
| basic_license_id | FK | 可选：下单时校验用的基础版授权 id，便于审计「先 9.9 后 19.9」 |
| use_count | int | 使用次数（可选） |
| device_fingerprint | string | 首次使用设备指纹（可选） |
| created_at | datetime | 创建时间 |

- 一次购买 19.9 生成一条记录、一个 token；**创建 AI 周卡订单时需校验请求方已拥有有效基础版**（见下）。

## 五、核心流程

### 5.1 免费档使用

- 用户打开网页，未激活基础版且无有效 AI token 时，视为免费档。
- 仅展示并允许点击「典型场景示例」中的「启保停控制」「延时启动」两项；点击后可将对应文案填入输入框，但**不开放「一键生成」按钮**或仅提示「升级基础版 9.9 元解锁本地生成」。
- 无需调用后端授权接口；前端根据「无基础版凭证、无有效 AI token」即可判定免费档。

### 5.2 基础版 9.9 元购买与使用

1. **下单**：前端请求「创建基础版订单」，后端生成订单（pending），返回订单号、金额、支付参数（或支付跳转 URL）。
2. **支付**：用户完成支付，支付渠道异步回调后端。
3. **回调处理**：后端验签、更新订单为 paid；生成一条 `basic_licenses`，生成唯一 `license_key`。
4. **交付**：成功页展示「您的永久授权码：XXXX」，提示用户保存。
5. **使用**：用户在「激活」入口输入授权码；前端发往后端校验，通过后前端写入本地存储；之后典型场景示例全部开放、本地生成入口开放，AI 入口展示「购买 19.9 AI 周卡（需已开通基础版）」。

可选：若希望「一码一设备」，可在首次校验时绑定设备指纹并写入 `basic_licenses`。

### 5.3 AI 周卡 19.9 元：依赖基础版方可购买与激活

1. **下单前置条件**  
   - 前端：仅当「已激活基础版」时展示并可点击「购买 AI 周卡」；未激活时展示「请先购买并激活基础版 9.9 元」。  
   - 后端：创建 AI 周卡订单时，请求体需携带**当前用户的基础版授权码**（或前端已保存的凭证）；后端校验该授权码在 `basic_licenses` 中存在且有效，校验通过才创建订单并返回支付参数；否则返回 400，例如 `{"code":"BASIC_REQUIRED","message":"请先购买并激活基础版"}`。

2. **支付与回调**  
   - 用户完成支付，支付渠道回调后端；后端生成一条 `ai_week_tokens`：`valid_from = now`，`valid_until = now + 7 天`，`token` 唯一随机串；可选写入 `basic_license_id` 便于审计。

3. **交付**  
   - 成功页展示「AI 周卡使用链接」与「二维码」，内容为 `https://你的域名/entry?t=xxx`，并提示 7 天内有效。

4. **使用**  
   - 用户通过链接或扫码打开页面，前端带 `t=xxx` 调用「校验 AI token」接口。  
   - **可选增强**：校验接口同时要求传入当前设备已激活的 `basic_license_key`（或由前端自动带上前端存储的基础版凭证）；后端校验：① token 有效且未过期 ② 该基础版授权有效。若 token 有效但未传或基础版无效，可返回 403，提示「请先在本设备激活基础版」。  
   - 若不做「同设备基础版」校验，仅依赖「下单时已校验基础版」也可接受，前端保证「仅基础版用户能看到并购买 19.9」即可。

5. **同一用户多次购买 AI 周卡**  
   - 建议首版仍采用「一单一线索」：每次购买生成新 token，独立 7 天；每次购买都需再次校验当时基础版已开通。

## 六、接口设计（概要）

### 6.1 订单与支付

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/order/create | POST | 创建订单；body: `{ productType: "basic" \| "ai_week", basicLicenseKey?: "xxx" }`；当 productType 为 ai_week 时，**必须传 basicLicenseKey**，后端校验该 key 有效后才创建订单，否则返回 400 BASIC_REQUIRED |
| /api/order/status | GET | 查询订单状态；query: orderId；返回 pay_status、若已支付则返回 license_key 或 token 信息 |
| （支付渠道）回调 | POST | 支付渠道回调 URL，后端验签、更新订单、发放 license 或 token |

### 6.2 授权与 token 校验

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/license/activate | POST | 激活基础版；body: `{ licenseKey: "xxx" }`；校验通过返回 success，可选返回加密后的前端凭证 |
| /api/license/validate | GET | 校验基础版是否有效；可带前端凭证或 licenseKey；返回 { valid: true/false } |
| /api/ai-token/validate | GET | 校验 AI 周卡 token；query: `t=xxx`；可选 body/header 带 basicLicenseKey 做「同设备基础版」校验；返回 { valid, validUntil } 或 401/403 |

### 6.3 前端使用方式

- **免费档**：不请求授权接口，仅根据「无基础版、无 AI token」展示 2 个场景 + 升级引导。  
- **基础版**：输入授权码后调 `/api/license/activate`；之后每次进入可调 `/api/license/validate` 或依赖本地存储。  
- **AI 周卡**：打开带 `t=xxx` 的链接时调 `/api/ai-token/validate?t=xxx`，成功则写入 AI 有效状态与过期时间；购买 19.9 前必须已激活基础版并传 basicLicenseKey 到 create order。

## 七、前端权限与展示逻辑

### 7.1 状态来源

- **免费档**：无基础版凭证且无有效 AI token → 免费档。  
- **基础版**：本地存储的「已激活」标记 + 可选服务端校验。  
- **AI 周卡**：URL 参数 `t` + 服务端校验结果 + 本地缓存的 validUntil。

### 7.2 展示规则（三档）

| 场景 | 典型场景示例 | 本地生成入口 | AI 智能生成入口 | 购买 19.9 入口 |
|------|--------------|--------------|------------------|----------------|
| 免费档 | 仅「启保停控制」「延时启动」可点，其余置灰/锁 | 不开放，提示购买 9.9 | 不开放 | 不展示或展示为「先购买基础版」 |
| 已购基础版、未购/过期 AI | 全部可点 | 开放 | 展示为「购买 19.9 AI 周卡（7 天）」 | 展示，可点击 |
| 基础版 + AI 有效期内 | 全部可点 | 开放 | 开放 | 可展示为「续费」或隐藏 |

### 7.3 典型场景示例与 constants 的对应

- 实现时在 `SCENARIOS` 中按**标题**或**固定索引**标记「免费可用」的 2 项（启保停控制、延时启动），其余根据当前档位决定是否可点击；生成逻辑仍复用现有「点击后 setScenarioText」，免费档仅允许对上述 2 项调用，其余点击不写入或提示升级。

## 八、安全与防滥用

- **token/授权码**：足够随机、不可预测；AI token 仅通过 HTTPS 传递。  
- **校验**：激活与校验均在后端完成；**AI 周卡下单强制校验基础版**，避免绕过前端直接调接口购买 19.9。  
- **限设备（可选）**：AI 周卡首次校验时记录设备指纹，同一 token 限制 1～2 台设备。  
- **防刷**：同一 IP/支付账号短时间大量下单可限流或人工审核。

---

## 九、与现有前端的衔接点

- 现有「典型场景示例」折叠区、`SCENARIOS` 列表保留；增加「档位判断」：免费档只允许 2 个场景（启保停控制、延时启动）可点击，其余展示锁或置灰。  
- 现有「本地生成 / AI 智能生成」切换保留；在其上层增加权限判断：无基础版不开放本地生成，无有效 AI token 不开放 AI 生成；**仅在有基础版时展示并允许发起 19.9 购买**。  
- 免登录（如 VITE_APP_SKIP_LOGIN）可继续用于开发/内测；正式环境改为「免费档默认 + 授权码激活 + 带 token 链接」即可，无需传统账号密码。

## 十、实施顺序建议

1. **后端**：订单表、基础版授权表、AI 周卡 token 表；创建订单时对 ai_week 校验 basicLicenseKey；支付回调发放 license/token；激活与校验接口。  
2. **支付**：对接支付渠道，完成 9.9 / 19.9 从创建订单到回调发放闭环。  
3. **前端**：免费档逻辑（仅 2 个场景可点）、购买页（9.9 / 19.9，19.9 仅基础版开通后可用）、支付跳转、成功页；入口权限与激活/校验调用。  
4. **联调与上线**：沙箱支付 → 真实小额 → 正式发布。
