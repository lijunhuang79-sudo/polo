# 网页两档收费 — 技术方案

> **业务方案**：基础版 9.9 元（永久本地生成）+ AI 周卡 19.9 元（7 天有效，凭二维码/链接使用）。本文档描述技术实现方案，不涉及具体代码实现细节。

---

## 一、方案概述

### 1.1 两档权益

| 档位 | 价格 | 权益 | 交付方式 |
|------|------|------|----------|
| 基础版 | 9.9 元 | 永久使用「本地生成」模式 | 支付成功后下发永久授权码/绑定账号 |
| AI 周卡 | 19.9 元 | 7 天内使用「AI 智能生成」模式 | 支付成功后生成 7 天有效 token，以链接+二维码形式交付 |

### 1.2 使用关系

- 未付费：仅可浏览或受限试用（具体策略可配置）。
- 仅购买 9.9：可使用本地生成；AI 智能生成入口提示「购买 AI 周卡」。
- 购买 9.9 + 19.9（或仅购买 19.9，若允许）：在 7 天内可使用 AI 智能生成；7 天后 AI 入口再次提示购买周卡。
- 同一用户可多次购买 19.9，每次获得新的 7 天周期（可设计为叠加或覆盖，见后文）。

### 1.3 技术目标

- 网页端完成支付、领取权益、凭链接/二维码登录。
- 后端完成订单记录、授权与 token 生成、校验与过期控制。
- 前端根据「基础版是否解锁」「AI token 是否有效」控制入口与能力。

## 三、数据模型（建议）

### 3.1 订单表 `orders`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | 订单号（可兼作业务订单号） |
| product_type | enum | `basic` \| `ai_week` |
| amount | decimal | 金额（9.9 / 19.9） |
| pay_status | enum | pending \| paid \| refunded \| failed |
| pay_channel | string | 支付渠道订单号 |
| pay_time | datetime | 支付成功时间 |
| buyer_open_id / buyer_id | string | 支付渠道用户标识（若需） |
| created_at | datetime | 创建时间 |

### 3.2 基础版授权表 `basic_licenses`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | |
| order_id | FK | 关联订单 |
| license_key | string | 永久授权码（唯一，可展示给用户保存） |
| used_at | datetime | 首次使用时间（可选） |
| created_at | datetime | 发放时间 |

- 一个订单对应一条 `basic` 记录，一条记录对应一个永久 `license_key`。

### 3.3 AI 周卡 token 表 `ai_week_tokens`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 主键 | |
| order_id | FK | 关联订单 |
| token | string | 唯一 token（放在链接中，或生成二维码内容） |
| valid_from | datetime | 生效时间（通常=创建时间） |
| valid_until | datetime | 过期时间（创建时间 + 7 天） |
| use_count | int | 使用次数（可选，用于限频或统计） |
| device_fingerprint | string | 首次使用设备指纹（可选，用于限设备） |
| created_at | datetime | 创建时间 |

- 一次购买 19.9 生成一条记录、一个 token；同一用户再次购买可再生成新 token（叠加或替换策略见下）。

## 四、核心流程

### 4.1 基础版 9.9 元购买与使用

1. **下单**  
   - 前端请求后端「创建基础版订单」，后端生成订单（pending），并返回订单号、金额、支付参数（或支付跳转 URL）。
2. **支付**  
   - 用户完成支付，支付渠道异步回调后端。
3. **回调处理**  
   - 后端校验签名、更新订单为 paid；生成一条 `basic_licenses`，生成唯一 `license_key`（如 UUID 或带校验的随机串）。
4. **交付**  
   - 前端轮询订单状态或由后端重定向到「成功页」；成功页展示「您的永久授权码：XXXX」，并提示用户保存（可复制按钮）。
5. **使用**  
   - 用户再次打开网页时，在「激活」入口输入该授权码；前端将授权码发往后端校验，校验通过则后端返回「基础版已激活」；前端将结果写入本地存储（如 localStorage 的 `basic_license` 或加密后的凭证），后续每次进入根据本地存储或短时缓存决定是否展示「本地生成」入口。

可选：若希望「一码一设备」，可在首次校验时绑定设备指纹并写入 `basic_licenses`。

### 4.2 AI 周卡 19.9 元购买与使用

1. **下单**  
   - 前端请求后端「创建 AI 周卡订单」，后端生成订单（pending），返回订单号、金额、支付参数。
2. **支付与回调**  
   - 同上，支付成功后回调。
3. **发放 token**  
   - 回调中生成一条 `ai_week_tokens`：`valid_from = now`，`valid_until = now + 7 天`，`token` 为唯一随机串（建议 32 位以上，带签名或查表校验）。
4. **交付**  
   - 成功页展示：  
     - 「AI 周卡使用链接」：`https://你的域名/entry?t=xxx`（xxx 为 token）；  
     - 「二维码」：内容即为该链接，方便手机扫码或分享。  
   - 并提示：链接/二维码 7 天内有效，请保存或扫码使用。
5. **使用**  
   - 用户通过「点击链接」或「扫码打开链接」进入 H5；  
   - 前端从 URL 读取 `t=xxx`，调用后端「校验 AI token」接口；  
   - 后端查表：token 存在、`valid_until > now` 则返回有效；可选：首次校验时写入 `device_fingerprint` 或限制同一 token 最多 N 台设备。  
   - 前端将「AI 有效」状态写入 sessionStorage 或短期 localStorage（建议与 token 一起存，过期时间取 `valid_until`）；  
   - 在有效期内展示「AI 智能生成」入口并允许使用；过期后隐藏或提示「请重新购买 AI 周卡」。

### 4.3 同一用户多次购买 AI 周卡

- **策略 A（叠加）**：每次购买生成新 token，多个 token 独立 7 天；前端可支持「输入多个 token」或「仅使用当前链接中的 token」。  
- **策略 B（覆盖/延长）**：同一买家标识（如 open_id）仅保留一个「当前有效」周期，新购买则把 `valid_until` 在原基础上 +7 天（需定义「同一用户」的识别方式）。  

建议首版采用 **策略 A**：一单一线索（一个链接/一个码），实现简单，逻辑清晰。

## 五、接口设计（概要）

### 5.1 订单与支付

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/order/create | POST | 创建订单；body: `{ productType: "basic" \| "ai_week" }`；返回订单号、金额、支付参数或跳转 URL |
| /api/order/status | GET | 查询订单状态；query: `orderId`；返回 pay_status、若已支付则返回 license_key 或 token 信息（用于成功页展示） |
| （支付渠道）回调 | POST | 支付渠道回调 URL，后端验签、更新订单、发放 license 或 token |

### 5.2 授权与 token 校验

| 接口 | 方法 | 说明 |
|------|------|------|
| /api/license/activate | POST | 激活基础版；body: `{ licenseKey: "xxx" }`；校验通过返回 success，可选返回加密后的前端凭证 |
| /api/license/validate | GET | 校验基础版是否有效；可带前端凭证或 licenseKey；返回 { valid: true/false } |
| /api/ai-token/validate | GET | 校验 AI 周卡 token；query: `t=xxx`；返回 { valid, validUntil } 或 401 |

### 5.3 前端使用方式

- 打开「带 token 的链接」时：先调 `GET /api/ai-token/validate?t=xxx`，成功则在本地写入 AI 有效状态与过期时间。  
- 进入「本地生成」前：若有本地保存的 license 凭证，可调 `GET /api/license/validate` 做一次校验（或定期刷新）；无凭证则引导购买 9.9 或输入授权码。

## 六、前端权限与展示逻辑

### 6.1 状态来源

- **基础版**：本地存储中的「已激活」标记 + 可选服务端校验结果。  
- **AI 周卡**：URL 参数 `t` + 服务端校验结果 + 本地缓存的 `validUntil`（避免每次请求都调接口）。

### 6.2 展示规则（建议）

| 场景 | 本地生成入口 | AI 智能生成入口 |
|------|--------------|------------------|
| 未购买基础版 | 不展示或展示为「购买 9.9 解锁」 | 不展示或展示为「先购买基础版」 |
| 已购买基础版、未购/过期 AI | 展示，正常使用 | 展示为「购买 19.9 AI 周卡（7 天）」 |
| 已购买基础版、AI 有效期内 | 展示 | 展示，正常使用 |

- 未付费时的「受限试用」若需要，可单独配置（如本地生成每日 N 次、AI 不可用等）。

## 七、安全与防滥用

- **token/授权码**：足够随机、不可预测；AI token 仅通过 HTTPS 链接传递，不在公开页展示完整列表。  
- **校验**：所有「激活」「校验」接口均在后端完成，不信任前端仅凭本地存储放行；AI 能力接口（若后端代理 AI）也应校验 token 或 session。  
- **限设备（可选）**：AI 周卡首次校验时记录设备指纹或 IP，同一 token 限制 1～2 台设备，超出则拒绝并提示「请在常用设备使用」。  
- **防刷**：同一 IP/同一支付账号短时间大量下单可限流或人工审核。

---

## 八、与现有前端的衔接点

- 现有「本地生成 / AI 智能生成」切换逻辑保留；在其上层增加「权限判断」：  
  - 无基础版授权时，本地生成入口不开放或跳购买页；  
  - 无有效 AI token 时，AI 智能生成不开放或跳「购买 AI 周卡」页。  
- 登录/免登录：当前若为免登录（如 VITE_APP_SKIP_LOGIN），可保留为开发/内测；正式环境关闭免登录，改为「授权码激活 + 带 token 链接」即可使用，无需传统账号密码（若不做账号体系）。

---

## 九、实施顺序建议

1. **后端**：订单表、基础版授权表、AI 周卡 token 表；创建订单、支付回调、发放 license/token；激活与校验接口。  
2. **支付**：对接一家支付渠道（如微信 Native/JSAPI 或支付宝），完成从「创建订单」到「回调发放」闭环。  
3. **前端**：购买页（基础版 / AI 周卡）、支付跳转、成功页（展示授权码或链接+二维码）；入口权限与「激活/校验」调用。  
4. **联调与上线**：沙箱支付 → 真实小额支付 → 正式发布。
