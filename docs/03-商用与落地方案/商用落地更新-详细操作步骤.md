# 商用落地更新 — 详细操作步骤

本文档说明：把「商用审查与优化」（后门关闭、免责声明、用户协议/隐私政策、错误边界、付费清单）部署到线上，并做一次自检。

---

## 第一步：在本机提交并推送代码

1. 打开 **PowerShell** 或 **CMD**，进入项目目录：
   ```powershell
   cd E:\Project\V2.13
   ```

2. 查看当前有哪些修改：
   ```powershell
   git status
   ```
   应能看到修改过的文件（如 `src/App.tsx`、`src/components/ErrorBoundary.tsx`、`docs/商用落地方案.md`、`docs/付费与商用落地清单.md` 等）。

3. 把所有要发布的改动加入暂存区：
   ```powershell
   git add src/App.tsx src/components/ErrorBoundary.tsx docs/商用落地方案.md docs/付费与商用落地清单.md
   ```
   若有其他改过的文件也要一起发布，可执行：
   ```powershell
   git add -A
   ```

4. 提交（写一条说明本次更新的备注）：
   ```powershell
   git commit -m "chore: 商用落地审查 - 后门关闭、免责声明、协议与隐私入口、ErrorBoundary、付费清单"
   ```

5. 推送到 GitHub：
   ```powershell
   git push origin main
   ```
   看到类似 “Writing objects... done” 即表示推送成功。若提示需要登录，按提示输入 GitHub 账号或 Token。

---

## 第二步：在服务器上拉取并重新部署前端

1. 用 SSH 登录你的服务器（把 `你的密码` 换成实际密码）：
   ```bash
   ssh root@213.111.157.18
   ```

2. 进入项目目录并拉取最新代码：
   ```bash
   cd /var/www/plc-sim/polo
   git pull origin main
   ```
   若提示 “Your local changes would be overwritten”，说明服务器上有未提交的修改，可先暂存再拉取：
   ```bash
   git stash
   git pull origin main
   git stash pop
   ```
   若不再需要服务器上的本地修改，也可直接丢弃：
   ```bash
   git reset --hard origin/main
   ```

3. 重新构建前端并发布到网站目录：
   ```bash
   npm run build
   mkdir -p /var/www/plc-sim/polo/frontend
   rsync -a --delete dist/ /var/www/plc-sim/polo/frontend/
   chown -R www-data:www-data /var/www/plc-sim/polo/frontend
   ```
   若 `rsync` 报错（如 unknown option），可改用：
   ```bash
   cp -r dist/* /var/www/plc-sim/polo/frontend/
   chown -R www-data:www-data /var/www/plc-sim/polo/frontend
   ```

4. （可选）若你之前已给部署脚本执行权限，也可用脚本一次性完成拉取+构建+发布：
   ```bash
   chmod +x /var/www/plc-sim/polo/deploy/scripts/deploy-polo.sh
   ./deploy/scripts/deploy-polo.sh
   ```

## 第三步：确认生产环境变量（避免后门被打开）

生产环境**不应**开启开发者后门或默认免登录。

1. 在服务器上检查前端构建时使用的环境变量。若你在服务器上执行的是 `npm run build`，且没有单独配置，则依赖仓库里的默认值；仓库里的 **env.production.example** 已为生产准备，不含后门开关。
2. 若你在服务器或 CI 里设置了环境变量，请确认：
   - **不要**设置 `VITE_APP_DEV_BACKDOOR=true`（否则后门仍可用）。
   - **不要**在生产环境设置 `VITE_APP_SKIP_LOGIN=true`（否则会免登录）。
   - 可设置 `VITE_APP_SKIP_LOGIN=false`（或不设置）。
3. 若构建时使用了 `.env.production`，在服务器上检查：
   ```bash
   cat /var/www/plc-sim/polo/.env.production
   ```
   确保其中没有 `VITE_APP_DEV_BACKDOOR=true`，且 `VITE_APP_SKIP_LOGIN=false`（或不写）。

## 第四步：在浏览器里自检

1. 打开网站：  
   **https://www.plc-sim.com**  
   输入 Nginx Basic Auth 的账号和密码（若已配置）。

2. 再输入**应用内**的调试密钥登录（若需要）。

3. 按下面清单逐项检查：

   - [ ] **页脚版本号**  
     应显示 **v2.0** 及**构建时间**。

   - [ ] **页脚免责声明**  
     有一行：「本产品仅供教学/方案参考，不替代实际 PLC 工程验证与安全认证。」

   - [ ] **用户协议与隐私政策**  
     页脚有「用户服务协议」「隐私政策」链接，点击后能弹出弹窗并显示占位内容，可点击「关闭」收起。

   - [ ] **生产环境下后门已关闭**（可选验证）  
     使用**生产构建**的页面（即线上 https://www.plc-sim.com），在登录框输入 `a` 并提交，应**无法**以开发者后门登录（应提示「身份验证失败」）。  
     （若你在服务器上误设了 `VITE_APP_DEV_BACKDOOR=true`，则后门仍可用，需按第三步改回。）

   - [ ] **错误边界**  
     正常使用无明显异常即可；错误边界会在子组件报错时显示「功能加载异常」和刷新按钮，无需特意制造错误。

4. 若页面仍是旧版，可**强制刷新**：  
   **Ctrl + F5**（Windows）或 **Cmd + Shift + R**（Mac）。

## 第五步：后续要做的（可选）

- **用户协议与隐私政策**：当前为占位文案，正式对外前请由法务审阅并替换为完整版（可在仓库或后台 CMS 中维护正文，前端链接不变即可）。
- **付费与计费**：按 **docs/付费与商用落地清单.md** 分阶段实施（用户体系 → 账户与流水 → 支付渠道 → 前端余额与充值入口）。
- **自动化部署**：若已配置 GitHub Actions，下次只需在本机 **git push origin main**，服务器会自动拉取并部署（需保证 Actions 中使用的脚本会执行 `npm run build` 并将 `dist/` 同步到 `frontend/`）。
