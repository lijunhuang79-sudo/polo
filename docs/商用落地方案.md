# PLC 编程仿真器 — 商用落地方案

> 从资深软件工程师视角，列出商用前必须解决的问题与实施建议。  
> 基于当前代码库：纯前端 SPA（Vite + React）、无后端、API Key 在前端、简单密码登录。

---

## 一、符合商用规范

### 1.1 法律与合规

| 项目 | 现状 | 建议 |
|------|------|------|
| **用户协议 / 服务条款** | 无 | 增加《用户服务协议》页面，明确：服务范围、免责（仿真结果仅供参考）、禁止用途、账号与数据规则、争议解决。可参考同类 SaaS 模板，由法务审阅。 |
| **隐私政策** | 无 | 增加《隐私政策》，说明：收集哪些数据（账号、使用记录、AI 请求内容等）、用途、存储地、保留期、第三方（如 AI 厂商）、用户权利（查阅/删除）。若面向欧盟需考虑 GDPR。 |
| **软件著作权 / 备案** | 未体现 | 申请软著；若在中国大陆提供在线服务，网站需 ICP 备案；若涉及支付需企业主体与相关资质。 |
| **工业/教育场景声明** | 无 | 在界面和文档中明确：「本产品仅供教学/方案参考，不替代实际 PLC 工程验证与安全认证。」降低责任风险。 |

### 1.2 当前需立即修正的合规问题

- **登录逻辑**：`password === 'a'` 为开发者后门，商用必须移除；改为正式用户体系（见下文）。
- **环境变量**：`VITE_APP_SKIP_LOGIN` 等仅用于开发；生产构建禁止默认免登录，并确保不把敏感配置打进前端包。

## 二、App 稳定运行

### 2.1 前端稳定性

- **React 错误边界**：为整页或关键模块（如仿真面板、AI 生成）包一层 `ErrorBoundary`，捕获渲染错误并展示友好提示，避免白屏。
- **请求与异步**：对所有 `fetch`（含 AI 调用）做超时、重试（可配置次数）、统一错误处理与用户提示；避免未捕获 Promise 导致静默失败。
- **状态与副作用**：仿真循环 `setInterval` 已在 `useEffect` 中清理，保持；确保所有异步或订阅在组件卸载时清理，避免内存泄漏与重复请求。

### 2.2 运行时与运维

- **健康检查**：若后续增加后端，提供 `/health` 等接口；前端可对关键 API 做简单可用性探测（不暴露敏感信息）。
- **日志与监控**：前端可上报匿名错误（如 Sentry）；后端记录请求量、失败率、延迟，便于发现故障与性能瓶颈。
- **部署与发布**：使用 CI/CD 做构建、测试、部署；生产环境关闭 source map 或仅内网可访问，避免泄露源码。

### 2.3 测试建议

- **单元测试**：对 `plcLogic`、`aiSolutionValidator` 等纯逻辑做单测，保证算法与校验稳定。
- **集成/ E2E**：对「登录 → 填写场景 → 本地生成 / AI 生成 → 导出」等主流程做少量关键路径 E2E，防止改坏核心流程。

## 三、API Key 安全方案（核心）

### 3.1 现状与风险

- 当前：用户在前端输入 API Key，存于 `localStorage`，由浏览器直接请求 DeepSeek / Gemini / OpenAI / Qwen。
- 风险：Key 暴露在客户端（可被开发者工具、脚本、打包产物分析拿到）；若用户共用或误配置，易被滥用或泄露；且无法做统一用量控制与审计。

### 3.2 推荐方案：后端代理 + 平台统一 Key（或用户自带 Key 加密托管）

**原则：所有第三方 API Key 只存在于服务端，前端永不接触真实 Key。**

```
┌─────────────┐     ① 请求(仅业务参数)     ┌─────────────┐     ② 用服务端 Key 调第三方    ┌─────────────┐
│   浏览器    │ ────────────────────────> │   你的后端   │ ───────────────────────────> │ DeepSeek等  │
│  (无 Key)  │ <──────────────────────── │ (存 Key/配额)│ <─────────────────────────── │   API       │
└─────────────┘     ③ 返回生成结果        └─────────────┘     ④ 返回结果                 └─────────────┘
```

**实施要点：**

1. **新增后端服务**（Node.js / Python 等均可）  
   - 提供接口例如：`POST /api/ai/generate`，入参：`{ model, prompt }` 或带 `userId`、`sceneText` 等。  
   - 后端从环境变量或密钥管理服务读取对应模型的 API Key，请求第三方，将结果返回前端。  
   - 不在任何响应体或日志中返回、记录 API Key。

2. **前端改造**  
   - 移除所有「用户输入 API Key」的 UI 与 `localStorage` 读写。  
   - 前端只调自己的后端 `/api/ai/generate`（或 `/api/ai/test` 做连通性测试）；如需鉴权，则带登录态 Token（见下一节）。

3. **可选：用户自带 Key 的「加密托管」**  
   - 若希望支持用户绑定自己的 Key（例如用自家 DeepSeek 账号）：  
     - 用户在前端输入 Key 后，通过 HTTPS 提交到后端；  
     - 后端用仅服务端持有的密钥对 Key 加密后落库，并只返回一个「已绑定」状态给前端；  
     - 后续请求由后端根据当前用户解密后调用第三方，前端仍不接触 Key。  
   - 需在隐私政策中说明「我们会加密存储您提供的 API Key 并仅用于为您调用 AI 服务」。

4. **配额与限流**  
   - 在后端按用户/套餐做调用次数或 token 用量限制，防止滥用与成本失控。  
   - 对未登录或匿名访问可拒绝 AI 能力或严格限流。

### 3.3 简要对照

| 项目 | 当前 | 目标 |
|------|------|------|
| Key 存放位置 | 浏览器 localStorage | 仅服务端（环境变量或密钥管理） |
| 前端是否接触 Key | 是 | 否 |
| 请求路径 | 浏览器 → 第三方 API | 浏览器 → 你的后端 → 第三方 API |
| 用量与审计 | 无法控制 | 后端统一限流、计费、审计 |

## 四、收费平台与计费方式

### 4.1 推荐：预充值 + 按次/按量扣费

- **模型**：用户先充值（余额或「点数」），使用 AI 生成时按次或按 token 扣减；本地生成可免费或单独定价。  
- **优点**：实现简单、易于理解、便于与现有「调用后端一次 = 扣一次费」对齐；可做活动（如首充送、包月套餐）。

### 4.2 支付渠道选择（简便、合规）

| 地区/用户 | 推荐方式 | 说明 |
|-----------|----------|------|
| 中国大陆 | 微信支付 + 支付宝 | 需企业主体、签约商户；可接官方 API 或通过 Ping++、BeeCloud 等聚合支付简化对接。 |
| 海外 / 国际卡 | Stripe | 支持订阅与一次性支付，文档完善，合规性好；若仅国内用户可暂不接。 |

**简便落地建议：**  
- 先接一种（如支付宝或微信），使用支付服务商提供的「下单 → 跳转/扫码支付 → 异步回调」流程。  
- 后端在「支付成功」回调中给用户账户增加余额或点数，并写流水表；AI 调用时扣减并校验余额。

### 4.3 后端需具备的能力（与收费相关）

- **用户与登录**：账号体系（注册/登录/找回密码），登录后颁发 Token（如 JWT），前端请求 AI/支付时携带。  
- **账户与计费**：用户表、余额/点数表、流水表；扣费时事务内「扣余额 + 记流水」；支付回调幂等处理，避免重复加钱。  
- **套餐与定价**：可先简单固定「每次 AI 生成 = N 元或 M 点」；后续再做包月、包年、不同模型不同价等。

### 4.4 定价思路（示例）

- 本地生成：免费。  
- AI 生成：按次（如 0.1 元/次）或按 token（与成本挂钩）；新用户送若干次免费额度以转化。  
- 导出报告等可设为免费或计入套餐，便于产品差异化。

## 五、其他必备与建议项

### 5.1 用户系统与认证

- 用你现有的「登录」仅作开发/内测；商用需：  
  - 注册（手机/邮箱 + 验证码或密码）、登录、登出、忘记密码。  
  - 使用 JWT 或 Session，前端请求头带 Token，后端校验后再执行 AI 与扣费。  
- 若先做 MVP，可用「手机号 + 验证码」登录，由云厂商（阿里云、腾讯云）提供短信验证码服务。

### 5.2 安全与风控

- **HTTPS**：全站 HTTPS，支付与 API 全部走 HTTPS。  
- **防刷与限流**：登录/注册/发送验证码做频率限制；AI 接口按用户限流，防止单用户拖垮服务。  
- **敏感操作**：支付、绑定手机/邮箱等可加二次验证或验证码。

### 5.3 数据与隐私

- **最小收集**：只收集业务必需数据（账号、使用记录、为生成而上传的文本）；不在前端或日志中记录 API Key。  
- **数据留存**：明确场景描述、生成结果等保留时长，到期删除或匿名化；在隐私政策中写明。  
- **导出与删除**：提供「导出我的数据」「注销账号」入口，满足合规与用户信任。

### 5.4 产品与体验

- **权限与功能开关**：未登录可仅使用本地生成；AI 与导出等需登录且余额充足再开放。  
- **提示与兜底**：AI 失败、余额不足、网络错误时，给出明确提示与下一步操作建议。  
- **帮助与反馈**：提供使用说明、常见问题、联系/反馈渠道，便于支持与迭代。

### 5.5 技术债与可维护性

- **配置外置**：API 根地址、功能开关、计费参数等放后端或配置中心，避免改前端才能调价/关功能。  
- **文档**：接口文档（如 OpenAPI）、部署文档、运行与监控说明，便于协作与运维。

## 六、实施优先级建议

| 阶段 | 内容 | 目的 |
|------|------|------|
| P0 | 后端代理 AI 请求 + 移除前端 API Key | 安全合规、为计费打基础 |
| P0 | 正式用户体系（注册/登录/Token） | 识别用户、计费与权限 |
| P1 | 用户协议与隐私政策页面 + 登录前勾选 | 合规 |
| P1 | 充值 + 按次扣费 + 一种支付渠道 | 商业闭环 |
| P2 | 错误边界、请求超时与重试、简单监控 | 稳定性 |
| P2 | 限流、防刷、日志与审计 | 安全与运维 |
| P3 | 套餐/定价策略、数据导出与账号注销 | 体验与合规深化 |

---

## 七、小结

- **商用规范**：补齐用户协议、隐私政策、免责声明，去掉硬编码后门，生产环境严格区分配置。  
- **稳定运行**：错误边界、统一请求处理、基础监控与测试，配合后端健康检查与 CI/CD。  
- **API Key**：必须改为「后端代理 + 服务端持 Key」，前端不再接触 Key；可选支持用户 Key 加密托管。  
- **收费**：预充值 + 按次/按量扣费，先接一种支付（如支付宝/微信），后端做账户与流水。  
- **补充**：正式用户系统、HTTPS、限流防刷、数据最小化与留存策略、帮助与反馈，按优先级分阶段落地。

按上述顺序推进，可在满足合规与安全的前提下，逐步将当前应用升级为可商用的 SaaS 产品。
